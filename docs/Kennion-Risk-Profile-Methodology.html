Kennion Risk Profile Methodology

How the Kennion website turns an uploaded census into a group-level risk score and risk-tier classification.

Prepared for actuarial review · Document version 1.0 · Source of truth: Kennion website production code base.

1. Purpose of this document

This document describes, end-to-end, how the Kennion website processes a customer-uploaded employee census and arrives at the group-level composite risk score and risk-tier classification (Preferred / Standard / High) that is shown to the customer and used in their proposal narrative. It is intended as a reference for the actuary so the demographic risk inputs and the qualification thresholds can be reviewed, challenged, or updated.

Note: the rate a customer is quoted on each plan (EE / EC / ES / FAM monthly premium) is computed by a separate engine that uses the actuary's Excel rater (age curve, area factors, plan base PMPM, trend, tier multipliers) and is described separately. The methodology in this document covers only the demographic risk profile / risk score shown on the proposal — i.e., the qualification signal, not the price.

2. What the customer uploads

The customer downloads a CSV template from the Kennion site and uploads it back populated with their employee + dependent roster. The template is strict — exactly six columns, named exactly as below — and the parser rejects the upload if any required column is missing or any extra column is present.

ColumnRequired valuesUsed for
First NameFree textIdentification only — not used in risk math
Last NameFree textIdentification only — not used in risk math
TypeEE (employee), SP (spouse), CH (child)Counts & dependency ratio; all members count equally in the risk score
Date of BirthAccepted: YYYY-MM-DD, M/D/YYYY, M-D-YYYYAge (used in age band lookup)
GenderMale / M or Female / FSelects male vs. female factor in the risk lookup
Zip Code5-digit US zipRating-area inference (rate engine, not risk score)

Sample template row: John,Smith,EE,3/15/1985,Male,30301.

3. Sensitive-data handling (before any processing)

The very first thing the parse endpoint does — before AI cleaning, before validation, before storage — is run the parsed CSV through a header-pattern scrubber that drops any column whose header matches a sensitive pattern. Dropped columns never reach the AI cleaner, the database, the session table, the response payload, or the request log.

The scrubbed patterns include:

  • Social Security number (SSN, "social security")
  • Tax ID, EIN, federal ID
  • Credit card number, CVV
  • Driver license number, passport
  • Bank account, routing number

The Kennion system stores no SSN or other tax-ID anywhere in its database and does not echo such fields back in any API response. The only place an SSN ever enters the system is the proposal-acceptance form, where the last 4 digits are entered by the prospective member, transmitted in the acceptance email body to Kennion staff, and immediately discarded — there is no DB write and no log line.

4. Column normalization (AI-assisted)

After the strict-template column check and PHI scrub, the cleaned headers and rows are passed to an LLM-backed cleaner whose only job is to normalize the values into a canonical schema:

  • Coerce Date of Birth into a single ISO-style date.
  • Map Type values to EE / SP / CH.
  • Map Gender values to Male / Female.
  • Trim & case-normalize names and zip codes.

The cleaner does not invent data, does not change the shape of the schema, and does not see any column the PHI scrubber removed. The output is shown back to the customer for confirmation before it is written to the database.

5. Storage

Confirmed rows are written to the census_entries table in PostgreSQL, one row per person, with a foreign key to the customer's groups row. The schema (in shared/schema.ts) holds: first name, last name, date of birth, gender, zip code, relationship — and nothing else from the original CSV.

6. Computing the group risk score

6.1 Step 1 — Per-person age

For each row in the census, age is computed as floor((today − dob) / 365.25 days). Rows with unparseable dates of birth, or with age ≤ 0 or age ≥ 120, are excluded from the score. Their absence is detected by the audit checks in section 8.

6.2 Step 2 — Age band

The age is mapped to one of fifteen 5-year bands:

0–4, 5–9, 10–14, 15–19, 20–24, 25–29, 30–34, 35–39, 40–44, 45–49, 50–54, 55–59, 60–64, 65–69, 70+.

6.3 Step 3 — Per-person demographic factor

The (age band, gender) pair indexes a fixed lookup table — the Demographic Risk Table — which returns a single numeric factor per person. The current production table is reproduced below in full. This is the table the actuary is asked to review, validate, or replace.

Fallback if an age band somehow does not resolve (it always should): Female = 0.97, Male = 0.80.

6.4 Step 4 — Group composite risk score

The group-level score is a simple unweighted arithmetic mean across all valid census members (employees, spouses, children — all weighted equally):

group_risk_score = ( Σ person_factor[i] ) / N

where:
  person_factor[i] = DemographicRiskTable[ age_band(i) ][ gender(i) ]
  N                = number of census members with valid DOB & gender

The result is rounded to 2 decimal places before tier classification.

Worked example. Group of three: a 45-year-old female, a 52-year-old male, a 38-year-old female. Per-person factors: 1.03, 1.20, 0.81. Composite = (1.03 + 1.20 + 0.81) / 3 = 1.013 → rounded to 1.01 → Standard Risk (per section 7).

7. Risk-tier classification

The composite score maps to one of three tiers using the cutoffs below. These cutoffs are flagged in code as "DO NOT MODIFY without approval" — they are deliberately the actuary-controlled control surface.

TierComposite scoreCustomer message (high level)
Preferred Risk < 1.00 Strong fit for the Kennion (RBP / self-funded) program; expected to see the largest savings vs. fully-insured.
Standard Risk 1.00 – 1.49 Good fit for the Kennion program; meaningful savings expected vs. fully-insured.
High Risk ≥ 1.50 Higher-utilization profile; the proposal narrative recommends evaluating fully-insured alongside the Kennion option and explains what factors drove the higher score.

8. Internal audit checks (every proposal)

Every time a group is analyzed, the system runs a set of self-consistency checks after the score is computed and before the result is shown to the customer. These exist to catch a bad upload, a partial parse, or a bug in the math. Any failure is surfaced in the admin UI as a hard error.

CheckWhat it verifies
Total Lives Match Census row count equals (a) the sum of EE + SP + CH and (b) the total of the age-band distribution table built downstream. Catches rows silently dropped due to a bad DOB.
Gender Total Match Sum of females + males equals the age-band table total. Catches rows with an unrecognised gender value.
Gender Breakdown Match Female and male counts independently match between the census and the age-band distribution.
Risk Score Recomputation Independently recomputes the composite score from the age-band × gender distribution and verifies it matches the stored score within 0.01.
Risk Tier Categorization Verifies the assigned tier is the one the published cutoffs would yield for the stored score.

If all checks pass, the report is marked audit-passed. The admin UI also exposes a per-group "score audit" panel that re-runs the same logic on demand and lets staff review the math person-by-person. There is also a short LLM-based "second-opinion" review that reads the demographic distribution and the computed score, and writes 2–3 sentences confirming the score is consistent with the profile. That LLM review is a sanity-check overlay only — it never alters the score or the tier.

9. What the customer (and Kennion staff) see

The output of the analysis surfaces in two places:

  • Proposal PDF. Includes total lives, employee count, dependency ratio, average employee age, age-band × gender distribution, composite risk score, risk tier, and a short narrative tailored to the tier (Preferred & Standard emphasize Kennion-program fit; High recommends comparing to fully-insured).
  • Admin dashboard. The same data plus the audit-check results, the per-person factor breakdown, and the proposal-acceptance state for that group.

11. Notes & known characteristics

  • Equal weighting. The composite score is an unweighted average across employees, spouses, and children. It is not weighted by expected member-month exposure. (Easy to change to PMPM-weighted if the actuary prefers.)
  • Two distinct factor systems. The age-band × gender factors in this document drive the qualification tier. The pricing engine uses a separate, finer-grained age curve (one factor per age 0–81, 6:1 slope) sourced from the actuary's Excel rater. The two are intentionally decoupled — the tier is a "fit" signal, the rate is the actual price.
  • Cutoffs are stable until you say otherwise. Preferred < 1.00, Standard 1.00–1.49, High ≥ 1.50. Source-controlled in the code and called out as actuary-owned.
  • How to change the table. The Demographic Risk Table is the single change-point for the qualification math. Replacing the 15-row table or the three cutoffs is a one-file edit and a redeploy; everything downstream (audit checks, PDF, narrative) re-derives from the new numbers automatically.
